# plan-format — the self-contained plan + index `/wise-revise` writes

Read by `skills/wise-revise/SKILL.md` §6. Defines the per-plan file
schema and the `docs/plans/README.md` index. Every plan is written for a
**fresh-context executor that never saw the investigation session** — so
all context is inlined and nothing references "the analysis above".

The per-plan schema is the same `PLAN-*.md` shape `ticket-auto` /
`ticket-plan` emit and `/wise-implement-plan-auto` consumes (it reads
`## Tasks`, `## Decisions Made`, `## Assumptions`, `## Validation`), so a
written plan is directly runnable later. This skill writes the plan; it
**never runs it**.

## One file per finding

Path: `docs/plans/<NNN>-<slug>.md` — `<NNN>` zero-padded and
**monotonic** (continue above the highest existing number; never clobber
or renumber). `<slug>` is a short kebab-case summary of the fix.

```markdown
# PLAN-<NNN> — <Title>

## Source
- Scope: <paths/globs/component the finding came from>
- Found by: <lens> · leverage <(impact ÷ effort) × confidence>
- Commit: <SOURCE_SHA>   ← line numbers/excerpts below are valid at this SHA; re-verify if HEAD moved
- Evidence: <file:line>, <file:line> …

## Summary
<2–4 sentences: the problem, why it matters, what "done" looks like.>

## Assumptions
<Every assumption the executor must rely on; any access/uncertainty gap.>

## Decisions Made
<The deliberate choices baked into this plan and a one-line rationale
each — approach, what is deliberately NOT changed, conventions to follow
(with a concrete example from the codebase).>

## Current state
<The relevant code excerpt(s) showing the problem, exact paths, matching
the repo's naming. Inline enough that the executor need not hunt.>

## Tasks
<Ordered parallelizable WAVES. Each task: a one-line description, a
`Reuse: <existing asset path>` or `New: <what>` classification, and an SP
estimate (wise-estimation, ≤3 SP/task once decomposed). Wave N+1 starts
only after wave N is done — the implement phase dispatches one executor
per task and runs waves in sequence.>

Wave 1
- [ ] <task> — Reuse: <path> — 1 SP
- [ ] <task> — New: <what> — 2 SP
Wave 2
- [ ] <task> — Reuse: <path> — 0.5 SP

Total: <sum, rounded to nearest Fibonacci> SP

## Testing
<New tests to add and the existing test pattern/file to copy.>

## Validation
<The exact, copy-pasteable verification commands and their EXPECTED
output — one per gate. e.g.:>
- `npm run typecheck` → exits 0, no errors
- `npm test -- src/api/foo.test.ts` → all pass
- `npm run lint` → no new warnings in scope

## Stop conditions
<Hard STOPs: "If <reality diverges from an Assumption / Current state no
longer matches at HEAD / a Validation gate cannot be made to pass>,
halt and report instead of improvising or widening scope.">
```

### Rules for an executable plan

- **Inline everything.** Exact paths, real code excerpts, the
  convention-with-example. No "as discussed", no pointer back to this
  session.
- **Every step is verifiable.** Each gate in `## Validation` pairs a
  command with its expected output, so the executor knows it succeeded.
- **Bounded scope.** The plan fixes the one finding. `## Decisions Made`
  states what is deliberately left alone; `## Stop conditions` forbid
  improvising when reality diverges.
- **Drift-aware.** `## Source` stamps `SOURCE_SHA`; if HEAD has moved,
  the executor re-verifies the excerpts before editing.
- **No secrets.** Reference credential *type* + `file:line` only.

## The index — `docs/plans/README.md`

Created/refreshed after the plans. When it already exists, reconcile:
keep existing rows, append the new plans, leave prior numbering intact.

```markdown
# Improvement plans

Generated by `/wise-revise`. Each plan is self-contained and
execution-ready — drive one to a merged PR with
`/wise-workflow-run implement-plan-auto docs/plans/<NNN>-<slug>.md`, run it
verbatim on this branch with
`/wise-implement-plan-auto docs/plans/<NNN>-<slug>.md`, file it as a ticket
and let the `ticket-auto` workflow build it from the ticket, or hand it to
an engineer. Execution is yours to schedule; `/wise-revise` only plans.

## Recommended order
1. <NNN>-<slug> — <one-line> (no deps)
2. <NNN>-<slug> — <one-line> (after 1)

## Status
| # | Plan | Category | Leverage | SP | Status | Depends on |
|---|------|----------|----------|----|--------|------------|
| <NNN> | <slug> | <category> | <score> | <SP> | TODO | — |

## Noted, not planned
<Lower-leverage or low-confidence findings surfaced but not turned into
a plan this pass, with a one-line reason each — so a later pass need not
re-discover them.>
```

`Status` values are the executor's to update (`TODO` → `IN PROGRESS` →
`DONE`); `/wise-revise` writes them as `TODO`.
